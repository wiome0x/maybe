### 交易整理设计建议

结论：建议保留 `分类`、`交易方`、`标签` 三者分开，不建议在业务模型上直接合并。

原因：

- `分类` 用来回答“这笔钱属于什么类型”，适合做统计、预算、报表，例如：餐饮、交通、工资、股息。
- `交易方` 用来回答“这笔交易是和谁发生的”，适合做商户识别、聚合同一商家消费、按交易对象查询，例如：星巴克、京东、招商银行。
- `标签` 用来回答“这笔交易还有什么额外特征”，适合做横向补充和多维筛选，一个交易可以有多个标签，例如：报销、出差、家庭、订阅、年费。

因此更合适的关系是：

- 一个交易通常只有一个主分类。
- 一个交易通常只有一个交易方。
- 一个交易可以有多个标签。

如果合并后的影响：

- `交易方` 和 `标签` 合并：会把“固定识别对象”和“灵活备注维度”混在一起，后续自动识别、规则命中、商户聚合都会变差。
- `标签` 和 `分类` 合并：会把“唯一主归类”和“多选补充信息”混在一起，预算、报表、统计口径会变乱。
- 页面层面可以靠近，但数据模型不建议合并：设置页可以考虑做成一个“交易整理”总入口，下分 `分类 / 交易方 / 标签` 三个分区，降低理解成本，但底层仍保持独立。

### 交易整理说明

建议在页面上补充简短说明文案，并给出例子，降低用户学习成本。

可采用以下规则：

#### 1. 什么时候使用交易方

适用于“对手方明确、名称相对稳定”的对象。

示例：

- 在沃尔玛消费 128 元：交易方 = `沃尔玛`
- 支付 Spotify 月费：交易方 = `Spotify`
- 收到公司发薪：交易方 = `XX 科技有限公司`
- 偿还招商银行信用卡：交易方 = `招商银行`

建议说明文案：

`交易方表示这笔交易是和谁发生的，通常填写商户、平台、公司、银行或个人名称。适合用于同一商家归集、搜索和自动识别。`

#### 2. 什么时候使用标签

适用于“需要补充说明，但不适合作为主分类”的信息。

示例：

- 分类是 `餐饮`，标签可以是 `报销`、`客户招待`
- 分类是 `交通`，标签可以是 `出差`、`打车`
- 分类是 `购物`，标签可以是 `家庭`、`孩子`
- 分类是 `订阅服务`，标签可以是 `工作`、`年度扣费`

建议说明文案：

`标签用于补充交易特征，一笔交易可以添加多个标签。适合标记报销、出差、家庭、订阅、项目等横向维度。`

建议实例文案：

- `报销`：这笔支出后续可以报销
- `出差`：这笔支出属于出差场景
- `家庭`：这笔支出用于家庭共同开销
- `订阅`：这是一笔周期性订阅扣费
- `客户招待`：这笔支出用于客户沟通或商务接待

可用于页面提示的示例文案：

`例如：报销、出差、家庭、订阅。一笔交易可添加多个标签，用于补充说明交易场景。`

#### 3. 如何区分分类、交易方、标签

可以给用户一个简单判断顺序：

1. 先确定 `分类`：这笔钱属于餐饮、交通、工资还是投资？
2. 再确定 `交易方`：这笔交易是和谁发生的？
3. 最后补 `标签`：这笔交易是否还有报销、出差、家庭、订阅等附加属性？

示例：

- “在星巴克购买咖啡，用于客户沟通，可报销”
- 分类：`餐饮`
- 交易方：`星巴克`
- 标签：`客户招待`、`报销`

#### 4. 执行建议

- 导入或同步后，优先自动识别 `交易方`
- 基于规则或 AI 自动推荐 `分类`
- `标签` 允许用户手动补充，也可用规则批量打标
- 在新建/编辑交易时，为 `交易方` 和 `标签` 增加占位提示与示例

建议界面提示语：

- 交易方输入框：`例如：星巴克、京东、招商银行`
- 标签输入框：`例如：报销、出差、家庭、订阅；可添加多个`
